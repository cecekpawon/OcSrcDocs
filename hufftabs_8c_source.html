46/* NOTE - regenerated tables to use shorts instead of ints
47 * (all needed data can fit in 16 bits - see below)
48 *
49 * format 0xABCD
50 * A = length of codeword
51 * B = y value
52 * C = x value
53 * D = number of sign bits (0, 1, or 2)
54 *
55 * to read a CW, the code reads maxbits from the stream (dep. on
56 * table index), but doesn't remove them from the bitstream reader
57 * then it gets the correct CW by direct lookup into the table
58 * of length (2^maxbits) (more complicated for non-oneShot...)
59 * for CW's with hlen < maxbits, there are multiple entries in the
60 * table (extra bits are don't cares)
61 * the bitstream reader then "purges" (or removes) only the correct
62 * number of bits for the chosen CW
63 *
64 * entries starting with F are special: D (signbits) is maxbits,
65 * so the decoder always checks huffTableXX[0] first, gets the
66 * signbits, and reads that many bits from the bitstream
67 * (sometimes it takes > 1 read to get the value, so maxbits is
68 * can get updated by jumping to another value starting with 0xF)
69 * entries starting with 0 are also special: A = hlen = 0, rest of
70 * value is an offset to jump higher in the table (for tables of
71 * type loopNoLinbits or loopLinbits)
72 */
